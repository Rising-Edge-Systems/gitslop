## Codebase Patterns
- Tests live in `src/main/__tests__/*.test.ts` using vitest; run with `npm test` or `npx vitest run <file>`.
- Quality gates: `npm run typecheck` (node + web) and `npm test`. Both must be green before committing.
- Main-process code uses Node built-ins freely (crypto, http) — avoid new deps.
- Main process entry: `src/main/index.ts`. New main-process modules should be siblings of index.ts.
- Fixed loopback OAuth redirect: `http://localhost:47823/callback` (see `src/main/gitlab-oauth.ts` `LOOPBACK_PORT`). Do not change — it is baked into user-registered GitLab OAuth apps.
- GitLab auth headers differ by account type: PAT uses `PRIVATE-TOKEN: <token>`; OAuth uses `Authorization: Bearer <access_token>`. `gitlabApiRequest()` in `src/main/index.ts` currently only sends `PRIVATE-TOKEN` — US-GL-003+ must branch on `account.authType` when calling any `/api/v4/*` endpoint.
- `IntegrationAccount` in `src/main/index.ts` has optional OAuth fields: `authType?: 'pat' | 'oauth'`, `refreshToken?` (encrypted), `expiresAt?` (ms), `clientId?`. Missing/undefined `authType` means legacy PAT account.
- Token encryption helpers live in `src/main/token-crypto.ts` (`encryptToken` / `decryptToken`). Both index.ts and gitlab-oauth.ts import them. The `plain:` prefix fallback lets tests run without Electron's `safeStorage`.
- `gitlabApiRequest(method, path, token, instanceUrl, body?, authType?)` in index.ts now branches on authType: `'pat'` sends `PRIVATE-TOKEN`; `'oauth'` sends `Authorization: Bearer`. Always pass the account's authType when calling it.
- GitLab IPC handlers should get their token via `getGitLabConfig()` (async) or `ensureFreshGitLabToken(account)` — never call `decryptToken(account.token)` directly, or you'll skip the refresh-token flow for OAuth accounts.
- `configureGitLabAccountStore({ read, write })` is called once at app startup in `app.whenReady` to wire the gitlab-oauth refresh flow to electron-store.

## 2026-04-10 - US-GL-003
- Implemented automatic OAuth refresh-token rotation for GitLab accounts.
- Files changed:
  - `src/main/token-crypto.ts` — NEW. Extracted `encryptToken`/`decryptToken` into a shared module so gitlab-oauth.ts can use them without a circular import with index.ts. Uses `safeStorage` when available; `plain:` fallback otherwise.
  - `src/main/gitlab-oauth.ts` — added `refreshGitLabToken(account)`, `ensureFreshGitLabToken(account)`, `configureGitLabAccountStore()`, and a `StoredGitLabAccount` type. refreshGitLabToken POSTs `grant_type=refresh_token` to `{instanceUrl}/oauth/token` and returns a cloned account with new encrypted tokens. ensureFreshGitLabToken returns the decrypted access token, transparently refreshing OAuth tokens that are within 60s of expiry and persisting the updated account via a configured accessor.
  - `src/main/index.ts` — removed the local `encryptToken`/`decryptToken` definitions (re-imported from token-crypto). Dropped the now-unused `safeStorage` import. Changed `gitlabApiRequest` to accept a 6th `authType` param and branch on `PRIVATE-TOKEN` vs `Authorization: Bearer`. Made `getGitLabConfig` async; it now calls `ensureFreshGitLabToken` and returns `{ token, instanceUrl, authType, expired? }`. Updated all gitlab IPC handlers (getUser, listMergeRequests, getMergeRequest, createMergeRequest) to await `getGitLabConfig` and pass `authType` through to `gitlabApiRequest`. Updated `gitlab:getAccounts` to call `ensureFreshGitLabToken` per account and mark the row with `error: 'GitLab session expired — please re-authorize'` when refresh fails. Added `configureGitLabAccountStore(...)` wiring in `app.whenReady`. Made `gitlab:isLoggedIn`, `gitlab:getInstanceUrl`, and `gitlab:parseRemote` read from the store directly (they don't need fresh tokens).
  - `src/main/__tests__/gitlab-oauth.test.ts` — added 6 tests covering the PAT pass-through, fresh-OAuth pass-through, missing-refresh-token failure path, refreshGitLabToken early returns for PAT / missing clientId, and configureGitLabAccountStore smoke.
- Quality: typecheck clean, all 127 tests pass (121 existing + 6 new).
- **Learnings for future iterations:**
  - `safeStorage` is not available in the vitest runtime — calling `encryptToken`/`decryptToken` on real payloads throws. Tests that need a stored token must use the `plain:` prefix so `decryptToken` short-circuits without touching safeStorage.
  - `configureGitLabAccountStore` registers a read/write accessor for the gitlab accounts list so ensureFreshGitLabToken can persist refreshed tokens without importing `electron-store` directly (which would create a circular dependency with index.ts).
  - GitLab refresh tokens are SINGLE-USE — every successful refresh returns a new refresh_token that invalidates the old one. Our refreshGitLabToken always writes the new refresh_token back into the account before returning.
  - `gitlab:isLoggedIn` intentionally does NOT call ensureFreshGitLabToken — that would hammer GitLab on every UI poll. It only checks whether any account is stored.
  - The `StoredGitLabAccount` interface in gitlab-oauth.ts is a structural alias for `IntegrationAccount` in index.ts; kept separate to avoid a circular import. If you add fields to one, update the other.
